# GGtem UI Copy Guide And Safe Autopilot Rules

Last updated: 2026-05-16

Purpose: GGtem 화면 문구를 일관된 톤으로 정리하고, 여러 에이전트가 동시에 작업할 때 자동으로 처리해도 되는 문서/문구 작업과 반드시 보고해야 하는 위험 작업을 구분한다.

Scope: 이 문서는 문서/가이드 기준이다. 기능 코드, `src/app/i18n.ts`, 어드민/지갑/권한 로직은 수정 대상이 아니다.

## 1. 유저 화면 문구 원칙

유저 화면은 설명보다 행동을 먼저 보여준다. 사용자가 지금 무엇을 눌러야 하는지, 다음에 어떤 상태가 되는지를 짧게 말한다.

- 버튼은 동사 중심으로 쓴다: `구매하기`, `충전 요청`, `출금 신청`, `채팅 열기`, `리뷰 남기기`.
- 카드 제목은 결과 중심으로 쓴다: `판매 중인 매물`, `확인 대기 주문`, `사용 가능 잔액`.
- 긴 설명은 위험, 돈, 보안, 취소 불가 행동에만 사용한다.
- 같은 의미의 단어는 한 화면 안에서 통일한다. 예: `충전`과 `입금`을 섞지 않는다. 유저 화면은 가능하면 `충전`, 관리자 화면은 `입금 처리`를 쓴다.
- 사용자를 탓하는 표현을 피한다. `잘못 입력했습니다`보다 `입력값을 확인해 주세요`가 낫다.
- 실패 문구는 복구 방법을 붙인다. `구매에 실패했습니다`보다 `잔액을 확인한 뒤 다시 구매해 주세요`가 낫다.
- 빈 상태는 다음 행동을 제안한다. `없음`보다 `아직 주문이 없습니다. 매물을 둘러보세요`가 낫다.
- 거래 단계는 현재 상태와 다음 행동을 함께 보여준다. `전달 중`만 쓰지 말고 `전달 중 · 판매자가 상품을 보내고 있습니다`처럼 보조 문구를 둔다.

권장 톤:

- 짧고 명확하다.
- 사용자가 바로 움직일 수 있다.
- 금액, 주문, 계정, 보안 관련 문구는 차분하고 구체적이다.
- 과장된 마케팅 문구보다 신뢰를 주는 운영 문구를 우선한다.

피해야 할 톤:

- `혁신적인`, `완벽한`, `최고의`처럼 검증하기 어려운 홍보형 표현.
- `당장`, `무조건`, `절대 안전`처럼 과도하게 단정하는 표현.
- 기능 설명을 버튼에 길게 넣는 방식.

## 2. 로그인/회원가입 문구 원칙

로그인/회원가입은 사용자가 막히기 쉬운 영역이다. 문구는 짧게 쓰되, 실패와 보안 안내는 원인을 이해할 수 있게 쓴다.

- 이메일, 비밀번호, 인증, 세션 만료, 권한 제한을 구분해서 말한다.
- 보안 실패 문구는 공격자에게 힌트를 과하게 주지 않는다.
- 이메일 인증이 필요한 경우 다음 행동을 직접 안내한다.
- 관리자 계정과 일반 사용자 계정의 로그인 위치가 다르면 명확히 안내한다.
- 비밀번호 재설정 문구는 불안감을 줄이고 절차를 안내한다.

예시:

| 상황 | 권장 문구 |
| --- | --- |
| 필수값 누락 | 이메일과 비밀번호를 입력해 주세요 |
| 인증 필요 | 이메일 인증 후 로그인할 수 있습니다 |
| 비밀번호 재설정 요청 | 재설정 링크를 이메일로 보냈습니다 |
| 세션 만료 | 다시 로그인해 주세요 |
| 관리자 로그인 차단 | 관리자 계정은 관리자 로그인 화면을 이용해 주세요 |
| 일반 계정 관리자 접근 | 이 계정은 관리자 화면에 접근할 수 없습니다 |

로그인/회원가입 버튼은 짧게 유지한다.

- `로그인`
- `회원가입`
- `인증 메일 다시 보내기`
- `비밀번호 재설정`
- `새 비밀번호 저장`

## 3. 어드민 화면 문구 원칙

어드민 화면은 상태와 다음 액션이 핵심이다. 운영자가 판단할 수 있도록 현재 상태, 위험도, 처리 버튼, 감사 로그 연결을 빠르게 보여준다.

- 제목은 관리 대상과 상태를 함께 드러낸다. 예: `출금 검토 대기`, `분쟁 주문`, `위험 신고`.
- 버튼은 운영 액션 그대로 쓴다. 예: `입금 승인`, `출금 거절`, `판매 제한 적용`, `SLA 해결`.
- 상태 배지는 짧게 쓴다. 예: `대기`, `검토 중`, `완료`, `거절`, `분쟁`, `제한`.
- 위험 액션에는 이유 입력, 영향 범위, 감사 로그 기록 여부를 안내한다.
- 어드민에게 사용자를 탓하는 표현보다 처리 기준을 보여준다.
- 숫자와 금액은 단위를 생략하지 않는다. 예: `10 USDT`, `3건`, `24시간`.
- 필터 문구는 짧고 반복 가능해야 한다. 예: `전체`, `대기`, `긴급`, `오늘`, `내보내기`.

어드민 위험 액션 안내 문구에는 다음 요소를 포함한다.

- 무엇이 바뀌는가.
- 누구에게 영향이 가는가.
- 되돌릴 수 있는가.
- 감사 로그에 남는가.
- 필요한 사유가 무엇인가.

예시:

`판매 제한을 적용하면 이 사용자는 새 판매글 등록과 판매 제안을 할 수 없습니다. 사유는 감사 로그에 남습니다.`

## 4. 다국어 문구 작성 원칙

GGtem은 글로벌 사용자 화면을 전제로 한다. 원문부터 번역하기 쉬운 구조로 써야 한다.

- 한 문장에 여러 의미를 넣지 않는다.
- 은어, 줄임말, 한국 서비스 관용구를 피한다.
- 변수 위치가 언어별로 바뀔 수 있으므로 문장 조립을 조심한다.
- 금액, 상태, 수량, 시간은 별도 값으로 다룬다.
- 번역 대상 문구와 운영 데이터는 분리한다.
- 버튼 문구는 언어별 길이 차이를 고려해 짧게 쓴다.
- 부정문보다 긍정적 행동문을 우선한다.

좋은 원문:

- `주문을 확인해 주세요`
- `판매자가 상품을 전달하고 있습니다`
- `출금 주소를 다시 확인해 주세요`

번역에 불리한 원문:

- `처리 ㄱㄱ`
- `문제없으면 눌러요`
- `이거 취소하면 복잡해져요`

다국어 키 설계 시 권장:

- 같은 문구는 재사용한다.
- 화면별 의미가 다르면 같은 한국어라도 키를 분리한다.
- 버튼, 상태, 도움말, 오류 문구를 구분한다.
- 관리자 내부 데이터 값은 번역 키로 강제로 바꾸지 않는다.

## 5. 어드민 관리 데이터와 번역 대상 구분

번역해야 할 것과 그대로 보존해야 할 것을 섞으면 운영 데이터가 망가질 수 있다.

번역 대상:

- 사용자에게 보이는 버튼, 라벨, 안내 문구.
- 상태 설명 문구.
- 빈 상태, 오류, 성공 안내.
- 공개 고객센터/FAQ/CMS 문구.
- 일반적인 카테고리 표시명.

번역하지 말아야 할 관리 데이터:

- 주문번호, 유저 ID, 지갑 주소, TXID.
- 관리자 메모 원문.
- 신고 내용 원문.
- 채팅 원문.
- 감사 로그 action, targetType, targetId.
- DB enum 값: `REQUESTED`, `COMPLETED`, `WITHDRAWAL_HOLD` 등.
- API action 값: `CONFIRM_DELIVERY`, `REJECT_WITHDRAWAL` 등.
- 보안/권한 식별자: role, permission, session token, hash.

표시는 번역하되 원본 값은 보존해야 하는 것:

- 주문 상태: 화면에는 `구매 확정 대기`, 데이터는 `BUYER_CONFIRM_PENDING`.
- 유저 상태: 화면에는 `출금 보류`, 데이터는 `WITHDRAWAL_HOLD`.
- 원장 유형: 화면에는 `구매 에스크로 잠금`, 데이터는 `BUYER_ESCROW_LOCKED`.
- 관리자 역할: 화면에는 `재무 담당`, 데이터는 `FINANCE`.

원칙: UI label은 바꿀 수 있지만 저장값, API action, 감사 로그 식별자는 바꾸지 않는다.

## 6. 자동운전 허용 작업

아래 작업은 docs 범위에서 자동으로 처리할 수 있다.

- `docs/**` 안의 새 문서 작성.
- 기존 문서의 오탈자, 목차, 표, 체크리스트 보완.
- 기능 코드 변경 없이 화면 문구 기준 제안.
- PPT 목차, 플로우, 스크린샷 목록, 운영 체크리스트 작성.
- 문구 개선안을 표 형태로 제안.
- 번역 대상/비대상 구분 문서화.
- 위험 작업을 실제 수행하지 않고 보고서에 정리.
- 기존 문서와 충돌하지 않는 범위의 링크/참조 추가.

자동운전 시 지켜야 할 기준:

- 기능 파일은 읽기만 한다.
- 변경 파일은 `docs/**`로 제한한다.
- 확실하지 않은 정책은 `확인 필요`로 표시한다.
- 돈, 권한, 보안, 개인정보는 실제 변경하지 않는다.
- 기존 코드의 깨진 문구를 발견해도 코드에서 바로 고치지 않고 문서에만 기록한다.

## 7. 자동운전 금지 작업

아래 작업은 자동으로 수행하지 않는다.

- `src/app/**`, `src/lib/**`, `prisma/**`, `public/**` 기능 코드 수정.
- `src/app/i18n.ts` 수정.
- 어드민 권한, 역할, 접근 제어 로직 변경.
- 지갑 잔액, 원장, 입금, 출금, 정산 로직 변경.
- 주문 상태 머신, 에스크로, 재고 잠금 로직 변경.
- 보안 파일, 환경변수, 암호화/해시/세션 로직 변경.
- DB schema, migration, generated Prisma 파일 수정.
- API action 이름, enum 값, 저장 데이터 구조 변경.
- 실제 사용자 데이터, 관리자 데이터, 로그, 지갑 주소 변경.
- 운영성 버튼을 연결하거나 위험 액션을 활성화.
- 대량 치환으로 의미가 달라질 수 있는 문구 수정.

금지 작업이 필요해 보이면 구현하지 말고 문서에 영향도와 확인 질문을 남긴다.

## 8. 보고만 해야 하는 위험 작업

다음 항목은 발견 즉시 보고하고, 자동 수정하지 않는다.

- 결제 PIN, 비밀번호, 세션, 이메일 인증, 비밀번호 재설정 관련 이상.
- 주문 금액, 수수료, 정산 금액, 원장 금액 계산 의심.
- 에스크로 잠금/해제, 구매요청 잠금, 출금 잠금 관련 의심.
- 입금 승인, 출금 완료, 환불, 판매자 릴리즈 액션의 문구와 실제 동작 불일치.
- 관리자 권한별 메뉴 접근 불일치.
- 감사 로그가 남아야 하는 액션인데 로그 기준이 불명확한 경우.
- 신고/분쟁/리뷰 숨김/판매 제한처럼 유저 권익에 영향을 주는 액션.
- 개인정보, 지갑 주소, TXID, 계정 인수 정보가 화면이나 문서에 그대로 노출될 위험.
- 다국어 번역 과정에서 DB enum/API action/감사 로그 식별자를 바꾸려는 요구.
- 기존 작업자가 수정 중인 기능 코드와 충돌할 가능성.

보고 형식 권장:

| 항목 | 위치 | 위험 | 제안 |
| --- | --- | --- | --- |
| 무엇을 발견했는가 | 파일/화면/문서 | 돈/권한/보안/개인정보 영향 | 수정 전 확인 질문 |

## 9. 예시 문구 변환표

| 기존/나쁜 문구 | 권장 문구 | 이유 |
| --- | --- | --- |
| 확인하세요 | 주문을 확인해 주세요 | 행동 대상이 명확함 |
| 처리하기 | 입금 승인 | 어드민 액션이 구체적임 |
| 실패했습니다 | 처리하지 못했습니다. 입력값을 확인해 주세요 | 복구 행동을 안내함 |
| 구매를 진행하시겠습니까? | 결제 PIN을 입력하고 구매하기 | 다음 행동이 명확함 |
| 취소하면 안 됩니다 | 취소 후에는 같은 주문으로 되돌릴 수 없습니다 | 위험을 구체적으로 설명함 |
| 돈이 부족합니다 | 사용 가능 잔액이 부족합니다. 충전 후 다시 시도해 주세요 | 잔액 종류와 해결책을 제시함 |
| 오류 | 요청을 처리하지 못했습니다 | 사용자에게 의미 있는 표현 |
| 상세보기 | 주문 상세 보기 | 이동 대상을 명확히 함 |
| 보내기 | 채팅 보내기 | 화면 맥락 없이도 이해됨 |
| 등록 | 판매글 등록 | 데이터 대상이 명확함 |
| 수정 | 변경사항 저장 | 결과 중심 |
| 삭제 | 판매글 숨기기 | 실제 동작이 soft hide라면 삭제 표현 금지 |
| 블랙 처리 | 판매 제한 적용 | 운영/권한 영향이 명확함 |
| 풀기 | 판매 제한 해제 | 대상 상태가 명확함 |
| 지급 | 판매자 정산 처리 | 돈 흐름이 명확함 |
| 환불 | 구매자에게 환불 | 수혜자를 명확히 함 |
| 승인 | 출금 완료 처리 | 어드민 금융 액션이 구체적임 |
| 반려 | 출금 거절 | 상태 변화가 명확함 |
| 신고 처리 | 신고 상태 변경 | 실제 처리 범위를 과장하지 않음 |
| 안전합니다 | GGtem 채팅 안에서 거래를 진행해 주세요 | 절대 안전 단정 회피 |

## 10. 다음 작업자가 확인할 체크리스트

문구 작업 전:

- 작업 범위가 `docs/**`인지 확인했다.
- 기능 코드와 `src/app/i18n.ts` 수정 요청이 아닌지 확인했다.
- 현재 작업이 돈, 권한, 보안, 개인정보에 영향을 주는지 판단했다.
- 기존 문서와 충돌하는지 확인했다.

문구 제안 시:

- 버튼은 짧고 행동형인가.
- 카드/단계 제목은 현재 상태나 다음 행동을 보여주는가.
- 위험/돈/보안 안내는 충분히 구체적인가.
- 유저 화면과 어드민 화면의 용어가 역할에 맞게 구분되었는가.
- 번역 대상과 저장 데이터가 분리되어 있는가.
- 원본 enum/API action/감사 로그 식별자를 바꾸지 않았는가.
- 실패 문구에 복구 방법이 있는가.
- 빈 상태 문구에 다음 행동이 있는가.

보고 전:

- 수정 파일이 `docs/ggtem-ui-copy-guide.md` 또는 다른 `docs/**` 문서뿐인지 확인했다.
- 위험 작업은 수행하지 않고 보고 항목으로 남겼다.
- 다음 작업자가 바로 사용할 수 있도록 예시와 금지 기준을 포함했다.
- 확인이 필요한 정책은 단정하지 않고 `확인 필요`로 표시했다.
