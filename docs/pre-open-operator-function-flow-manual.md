# GGtem 오픈 전 운영자 기능 흐름 매뉴얼

최종 업데이트: 2026-05-17

이 문서는 GGtem 오픈 전 운영자가 반드시 이해해야 하는 기능 흐름을 운영 관점에서 정리한 문서다. 범위는 사용자 거래 흐름, 어드민 등록물품 확인, 프리미엄 노출, 보안 점검, 검색 등록 상태 확인이다. 코드나 DB를 수정하기 위한 문서가 아니라, 실제 오픈 전 운영자가 어떤 화면을 보고 어떤 기준으로 판단해야 하는지를 설명한다.

## 1. 운영자가 먼저 이해해야 할 서비스 흐름

GGtem은 판매글만 올리는 게시판이 아니라 판매 수요와 구매 수요를 동시에 다루는 거래 플랫폼이다. 판매자는 `/my/listings/new`에서 게임머니, 아이템, 계정 판매글을 등록하고 구매자는 `/my/buy-requests/new`에서 원하는 게임, 서버, 수량, 단가를 지정해 구매글을 등록한다. 사용자는 `/listings`에서 판매글과 구매글을 탐색하고, 조건이 맞으면 즉시구매 또는 즉시판매 흐름으로 주문을 만든다.

운영자는 이 구조를 “등록물품 관리”와 “거래 자금 관리”로 나누어 보아야 한다. 등록물품 관리는 판매글과 구매글이 공개 마켓에 적절히 노출되고 있는지, 부적절한 내용이나 위험한 계정 거래 정보가 없는지, 프리미엄 노출이 정상적으로 적용되고 있는지 확인하는 업무다. 거래 자금 관리는 충전, 구매요청 잠금, 주문 에스크로, 판매자 정산, 출금, 환불, 분쟁 처리가 원장과 일치하는지 확인하는 업무다.

오픈 전에는 특히 등록물품 확인 화면과 프리미엄 화면을 매일 점검해야 한다. 판매글과 구매글은 실제 유입과 거래 전환의 첫 화면이므로, 여기에서 부적절한 글이 노출되거나 프리미엄 만료 상태가 잘못 보이면 서비스 신뢰도가 즉시 떨어진다.

## 2. 일일 운영 시작 체크 순서

운영자는 업무 시작 시 다음 순서로 상태를 확인한다.

| 순서 | 화면 | 확인 목적 | 이상 징후 |
| --- | --- | --- | --- |
| 1 | `/admin` | 운영 요약, 처리 대기, 위험 신호 확인 | 대기 건수 급증, SLA 지연 |
| 2 | `/admin/market-listings` | 판매글/구매글 공개 상태 확인 | 숨김/삭제 처리 증가, 환불 대상 구매글 증가 |
| 3 | `/admin/premium` | 프리미엄 노출, 만료 예정, 수익 원장 확인 | 만료 임박 다수, 수익 원장 누락 |
| 4 | `/admin/orders` | 주문 상태와 분쟁 가능 주문 확인 | 장기 지연 주문, 문제 신고 |
| 5 | `/admin/disputes` | 분쟁 판정 대상 확인 | 증빙 부족, 장기 미처리 |
| 6 | `/admin/finance` 또는 `/admin/withdrawals` | 충전/출금/정산 상태 확인 | TXID 누락, 출금 보류 급증 |
| 7 | `/admin/audit` | 관리자 조치 기록 확인 | 사유 없는 조치, 권한 이상 |

이 순서는 단순한 화면 순회가 아니라 거래 안정성 점검 순서다. 공개 노출 상태를 먼저 보고, 이어서 돈이 움직이는 흐름과 관리자 조치 이력을 확인해야 한다.

## 3. 어드민 등록물품 확인 운영

등록물품 확인 화면은 `/admin/market-listings`이며, 운영자 권한 그룹 기준으로 주문 운영자가 접근한다. 이 화면은 판매글과 구매글을 동시에 확인하는 운영 콘솔이다. 판매글은 `Listing`, 구매글은 `BuyRequest` 데이터 기준으로 조회되며, 화면 상단에는 판매 전체, 판매중, 숨김, 삭제 처리, 구매 전체, 구매중, 수락됨 상태가 요약된다.

운영자는 이 화면을 통해 공개 마켓에 어떤 글이 노출되는지, 어떤 글이 조치 대상인지, 구매글의 잠금 금액이 환불 대상인지 확인한다. 필터는 거래 방향, 상태, 품목, 검색어로 구성되어 있다. 검색어는 제목, ID, 유저, 게임, 서버 기준으로 사용한다. 운영 중에는 특정 유저 신고가 들어왔을 때 유저 이메일이나 표시 이름으로 검색하고, 특정 게임 이슈가 있을 때 게임명 또는 서버명으로 묶어서 확인한다.

### 3.1 판매글 상태 확인 기준

판매글은 `ACTIVE`, `PAUSED`, `SOLD_OUT`, `HIDDEN`, `REMOVED` 상태를 기준으로 운영한다. `ACTIVE`는 공개 판매중인 글이고, `PAUSED`는 판매자가 일시중지했거나 운영상 임시 중지된 글이다. `SOLD_OUT`은 재고가 소진된 상태이므로 운영자가 임의로 다시 `ACTIVE`로 되돌리는 대상이 아니다. `HIDDEN`은 공개 노출을 숨긴 상태이고, `REMOVED`는 삭제 처리에 가까운 강한 조치 상태다.

운영자가 판매글에 조치할 수 있는 상태는 숨김, 삭제 처리, 중지, 복구다. 조치 시 사유 입력이 필요하며, 사유는 감사 로그에 남는다. 운영자는 사유를 “부적절함”처럼 짧게 쓰지 말고, 사용자가 나중에 이의를 제기해도 판단 근거를 재구성할 수 있게 써야 한다.

권장 사유 예시는 다음과 같다.

| 조치 | 사용 기준 | 사유 작성 예 |
| --- | --- | --- |
| 숨김 | 내용 확인 전 공개 노출을 막아야 할 때 | “계정 회수 위험 신고 접수, 증빙 확인 전 임시 숨김” |
| 삭제 처리 | 명백한 금지 표현, 사기 의심, 반복 위반 | “외부 메신저 유도 문구 포함, 동일 위반 2회 확인” |
| 중지 | 판매자 요청 또는 재고/가격 오류 가능성 | “판매자 가격 오류 신고로 임시 중지” |
| 복구 | 검토 후 정상 글로 판단 | “신고 내용 불충분, 이미지/본문 정상 확인 후 복구” |

판매글 조치 후에는 반드시 해당 글의 감사 로그 링크를 확인한다. `/admin/market-listings` 목록과 상세 화면 모두 감사 로그로 이동할 수 있도록 연결되어 있다. 감사 로그에는 조치 전 상태, 조치 후 상태, 관리자 ID, 사유가 남아야 한다.

### 3.2 구매글 상태 확인 기준

구매글은 구매자가 원하는 가격과 수량을 등록하고, 필요한 금액을 잠근 수요 등록이다. 구매글 목록에서는 총액, 잠금 금액, 제안 수, 본문 이미지 수, 프리미엄 여부를 확인할 수 있다. 운영자가 특히 봐야 할 값은 `lockAmount`다. `ACTIVE` 상태이면서 잠금 금액이 0보다 크면 운영자 취소와 환불 조치가 가능한 상태로 표시된다.

구매글 취소+환불은 자금 흐름에 영향을 주는 조치이므로 판매글 숨김보다 더 보수적으로 처리한다. 현재 구조상 운영자 취소가 실행되면 구매글 상태가 `CANCELED`로 바뀌고, 구매글에 잠긴 금액은 구매자의 사용 가능 잔액과 출금 가능 잔액으로 되돌아간다. 동시에 구매요청 잠금 해제 원장과 사용 가능 잔액 복구 원장, 출금 가능 잔액 복구 원장이 기록된다.

운영자는 구매글 취소 전 다음을 확인한다.

| 확인 항목 | 운영 기준 |
| --- | --- |
| 구매글 상태 | `ACTIVE`인지 확인한다. 이미 수락, 완료, 취소된 글은 즉시 환불 대상이 아니다. |
| 잠금 금액 | `lockAmount`가 0보다 큰지 확인한다. 잠금 금액이 없으면 환불할 금액이 없다. |
| 제안/주문 연결 | 판매자 제안이나 주문 전환이 진행 중인지 확인한다. |
| 사용자 요청 근거 | 고객센터, 채팅, 신고, 운영 메모 등 취소 근거를 확보한다. |
| 사유 | 환불 사유를 감사 로그와 원장 메모로 재구성 가능하게 작성한다. |

## 4. 등록물품 상세 화면 운영

등록물품 상세 화면은 `/admin/market-listings/sell/[id]` 또는 `/admin/market-listings/buy/[id]` 구조다. 운영자는 상세 화면에서 제목, 본문, 이미지, 작성자, 이메일, 게임, 서버, 상세 서버, 거래 방식, 가격 기준 단위, 단가, 프리미엄 시작/만료, 수량 정보를 확인한다.

판매글 상세에서는 총 수량, 최소 수량, 구매 가능 수량, 잠금 수량, 판매 완료 수량을 확인한다. 잠금 수량이 있다는 것은 진행 중인 주문 또는 재고 잠금이 있을 수 있다는 의미이므로, 판매글을 강하게 조치하기 전 주문 화면과 함께 확인해야 한다. 구매글 상세에서는 요청 수량, 최소 수량, 잔여 수량, 만료, 총액, 잠금액을 확인한다. 잠금액은 운영자 환불 판단의 핵심 값이다.

본문 이미지는 단순 장식이 아니라 분쟁과 사기 판단의 증거가 될 수 있다. 계정 거래의 경우 이미지에 로그인 정보, 개인정보, 외부 연락처, 회수 유도 문구가 포함될 수 있으므로 운영자가 직접 확인해야 한다. 민감 정보가 보이면 글 숨김 또는 삭제 처리를 우선하고, 사용자에게 수정 요청을 남기는 방식으로 운영한다.

## 5. 프리미엄 노출 운영

프리미엄 관리 화면은 `/admin/premium`이다. 이 화면은 현재 노출 중인 프리미엄 글, 6시간 내 만료 예정 글, 최근 만료 글, 누적 프리미엄 수익을 보여준다. 판매글과 구매글 모두 프리미엄 대상이 될 수 있으며, 화면에서는 판매글/구매글, 카테고리, 상태, 게임, 서버, 소유자, 단가, 이용료, 기간, 남은 시간, 시작/만료 시간이 함께 표시된다.

프리미엄 노출은 단순 광고 기능이 아니라 GGtem의 초기 수익 모델과 유동성 확보 장치다. 판매자는 재고를 더 빨리 노출시키고, 구매자는 자신의 수요를 판매자에게 더 잘 보이게 할 수 있다. 따라서 운영자는 프리미엄 화면을 매일 확인해 결제된 노출이 정상 유지되는지, 만료 시간이 정상 표시되는지, 프리미엄 수익 원장이 누락되지 않았는지 봐야 한다.

운영 기준은 다음과 같다.

| 항목 | 운영자가 보는 값 | 판단 기준 |
| --- | --- | --- |
| 현재 노출 | activeItems | 노출 중인 글이 실제 관리 상세로 이동되는지 확인 |
| 6시간 내 만료 | expiringSoonCount | 만료 임박 글이 많은 경우 판매자 문의 대비 |
| 최근 만료 | expiredItems | 만료 후에도 공개 노출이 과도하게 유지되는지 확인 |
| 누적 수익 | revenueTotal | 프리미엄 결제 원장 합계와 대사 |
| 최근 결제 | recentLedgerEntries | `PREMIUM_PROMOTION_PURCHASED` 원장 기록 확인 |

프리미엄 수익은 지갑 원장에서 `PREMIUM_PROMOTION_PURCHASED`, `CREDIT`, `PLATFORM_REVENUE` 버킷으로 조회된다. 운영자는 프리미엄 결제 문의가 들어오면 `/admin/premium`에서 해당 글을 찾고, 이어서 `/admin/finance/ledger?type=PREMIUM_PROMOTION_PURCHASED`에서 원장 기록을 확인한다.

주의할 점은 프리미엄 만료가 거래글 자체의 정상성 보증을 의미하지 않는다는 것이다. 프리미엄 글도 신고, 외부 연락처 유도, 허위 가격, 계정 회수 위험이 있으면 일반 글과 동일하게 숨김 또는 삭제 처리할 수 있다. 다만 유료 노출이므로 조치 사유와 근거는 더 자세히 남겨야 한다.

## 6. 보안 운영 체크

오픈 전 보안 점검은 운영자가 매일 확인해야 할 기본 업무다. 현재 문서 기준 필수 보안 값은 `docs/production-security-checklist.md`에 정리되어 있으며, 운영자는 아래 항목을 서비스 오픈 전 반드시 확인한다.

| 영역 | 확인 항목 | 운영 기준 |
| --- | --- | --- |
| 도메인 | `ggtem.com`, `topofword.com` 분리 | 사용자 사이트와 어드민 사이트가 분리되어야 한다. |
| 어드민 로그인 | 비밀번호 후 이메일 OTP | 운영 계정은 OTP 실패 잠금과 재발송 쿨다운이 작동해야 한다. |
| 어드민 권한 | SUPER, ADMIN, FINANCE 등 최소 권한 | 테스트 관리자 계정은 비활성화 또는 삭제한다. |
| 금융 조치 | 비밀번호 재확인, TXID 필수 | 충전 승인과 출금 완료는 TXID 없이 처리하지 않는다. |
| 출금 | 결제 PIN, 주소 형식 검증 | TRC20/BEP20 주소 형식과 PIN 잠금을 확인한다. |
| 감사 로그 | 관리자 조치 기록 | 등록물품 조치, 환불, 출금, 권한 변경 후 로그를 확인한다. |
| 업로드 | 이미지 용량/형식 제한 | 고용량 또는 변조 이미지가 차단되는지 확인한다. |
| 비밀값 | 서버 전용 환경변수 | `DATABASE_URL`, `TELEGRAM_BOT_TOKEN`, 서비스 키를 브라우저에 노출하지 않는다. |

운영자가 직접 변경하면 안 되는 영역도 명확히 구분해야 한다. 지갑 잔액, 에스크로 금액, 원장 기록, 출금 상태, 관리자 권한은 화면의 정상 액션을 통해서만 처리한다. DB에서 직접 수정하거나 코드 배포 없이 임의로 금액을 맞추는 방식은 금지한다.

## 7. 검색 등록 및 공개 노출 상태

검색 등록 상태는 외부 검색엔진이 GGtem의 공개 페이지를 인식할 수 있는지를 확인하는 운영 항목이다. 현재 구현 기준으로 공개 사이트는 `GGITEM_BASE_URL` 값을 사용해 사이트맵과 robots 정보를 생성한다. 기본 기준 도메인은 `https://ggtem.com`이다.

현재 검색 관련 구현 상태는 다음과 같다.

| 항목 | 현재 상태 | 운영 의미 |
| --- | --- | --- |
| 사이트맵 | `/sitemap.xml` 제공 | 홈, 목록, 고객센터, 로그인, 회원가입 공개 경로 포함 |
| robots | `/robots.txt` 제공 | `/admin`, `/api`, `/my`, 인증/복구 경로 차단 |
| Naver 인증 | 메타 verification 값 존재 | 네이버 사이트 소유 확인용 메타가 레이아웃에 포함됨 |
| 기본 메타 | applicationName/title/description 존재 | 기본 서비스명과 설명 제공 |
| PWA manifest | `/manifest.webmanifest` 연결 | 모바일 설치와 기본 아이콘 노출 기반 |

오픈 전 운영자는 다음 순서로 검색 등록 상태를 확인한다.

1. `https://ggtem.com/robots.txt`가 열리는지 확인한다.
2. `https://ggtem.com/sitemap.xml`이 열리고 공개 경로가 포함되어 있는지 확인한다.
3. `robots.txt`에서 `/admin`, `/api`, `/my`가 차단되어 있는지 확인한다.
4. 네이버 서치어드바이저에서 사이트 소유 확인이 완료되는지 확인한다.
5. 구글 Search Console에 `https://ggtem.com/sitemap.xml`을 제출한다.
6. 검색엔진에 공개되면 안 되는 마이페이지, 지갑, 주문, 관리자 페이지가 검색 결과에 노출되지 않는지 주기적으로 확인한다.

검색 등록은 마케팅 업무처럼 보이지만 보안 업무와도 연결된다. 관리자 도메인과 개인 거래 페이지가 검색 결과에 노출되면 운영 리스크가 커진다. 따라서 검색 등록 담당자는 사이트맵 제출뿐 아니라 robots 차단 경로가 실제로 적용되는지 함께 확인해야 한다.

## 8. 오픈 전 등록물품 샘플 점검

오픈 전에는 실제 거래를 시작하기 전에 샘플 판매글과 샘플 구매글을 만들어 흐름을 점검한다. 이때 운영자는 공개 마켓에서 보이는 화면과 어드민에서 보이는 화면이 같은 데이터를 가리키는지 확인한다.

샘플 점검 순서는 다음과 같다.

| 단계 | 사용자 화면 | 어드민 확인 |
| --- | --- | --- |
| 판매글 등록 | `/my/listings/new` | `/admin/market-listings?mode=SELL`에서 노출 확인 |
| 판매글 상세 | `/listings/[listingId]` | 상세 화면의 가격/수량/이미지 일치 확인 |
| 구매글 등록 | `/my/buy-requests/new` | `/admin/market-listings?mode=BUY`에서 잠금 금액 확인 |
| 프리미엄 적용 | 등록/홍보 옵션 | `/admin/premium`에서 노출/만료/수익 확인 |
| 운영 조치 | 해당 없음 | 숨김, 복구, 구매글 취소+환불 후 감사 로그 확인 |
| 검색 상태 | `/sitemap.xml`, `/robots.txt` | 공개/비공개 경로 분리 확인 |

샘플 점검에 사용한 계정과 데이터는 실제 오픈 전 삭제 또는 비활성화해야 한다. 특히 프리미엄 결제, 구매요청 잠금, 출금, 충전 승인 테스트는 원장에 흔적이 남기 때문에 테스트 데이터 여부를 운영 메모에 남겨야 한다.

## 9. 운영자 판단 원칙

운영자는 빠른 처리보다 추적 가능성을 우선해야 한다. GGtem은 자금 잠금, 에스크로, 원장, 프리미엄 수익, 감사 로그가 연결되어 있으므로 한 화면에서 버튼을 누른 결과가 여러 데이터 흐름에 영향을 준다. 판매글 숨김은 공개 노출에 영향을 주고, 구매글 취소+환불은 사용자 잔액과 원장에 영향을 주며, 프리미엄 조치는 유료 노출과 매출 신뢰에 영향을 준다.

따라서 운영자는 다음 원칙을 따른다.

- 금액이 움직이는 조치는 사유와 근거를 남긴 뒤 처리한다.
- 판매글 또는 구매글 조치 후 감사 로그를 확인한다.
- 프리미엄 글은 유료 노출이지만 위반 글이면 일반 글과 동일하게 조치한다.
- 검색엔진에는 공개 페이지와 마케팅 페이지까지만 노출되도록 한다.
- 관리자 페이지, API, 마이페이지, 지갑, 주문, 인증 복구 경로는 검색 노출 대상이 아니다.
- 코드나 DB를 직접 수정하지 않고, 운영 화면의 정상 액션으로만 처리한다.

## 10. 남은 운영 문서 작업 후보

이 문서는 오픈 전 기능 흐름과 등록물품 운영을 중심으로 작성했다. 다음 문서는 별도 작업으로 이어서 정리하는 것이 좋다.

| 문서 후보 | 필요 이유 |
| --- | --- |
| 충전/출금 상세 운영 매뉴얼 | TXID, 체인, 출금 수수료, 반려/완료 기준을 별도 문서로 분리 |
| 분쟁 판정 매뉴얼 | 구매자 환불/판매자 지급 판단 기준과 증빙 체크리스트 필요 |
| 관리자 권한별 업무표 | CS, FINANCE, ADMIN, SUPER 권한별 가능한 조치 정리 필요 |

| 프리미엄 상품 정책 문서 | 유료 노출 환불 기준, 만료 안내, 위반 글 조치 기준 필요 |
